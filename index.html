Welcome to Example API

Example is a powerful REST API that enables user management, request authentication, and building amazing applications. This documentation will guide you through all the features.

Base URL

https://api.example.com/v1

Features

  • RESTful architecture with intuitive endpoints
  • Secure authentication with API keys and OAuth 2.0
  • Comprehensive error handling and status codes
  • Rate limiting ensuring fair usage

Your First Request

Here's a simple example of making a GET request to fetch users: 

curl -X GET "https://api.example.com/v1/users" \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json"

You'll receive a JSON response with user data. Check the GET /users endpoint for more details.

Quick Start Guide

Get up and running with Example API in minutes. Follow these steps to make your first successful API call.

Step 1: Get an API Key

Sign up for an account and generate an API key from your dashboard. Your API key is used to authenticate all requests.

Step 2: Install Dependencies

You can use any HTTP client. Here's an example using JavaScript with fetch: 

const API_KEY = 'your_api_key_here';
const BASE_URL = 'https://api.example.com/v1';

fetch(`${BASE_URL}/users`, {
  headers: {
    'Authorization': `Bearer ${API_KEY}`,
    'Content-Type': 'application/json'
  }
})
.then(response => response.json())
.then(data => console.log(data));

Step 3: Handle Responses

All responses are returned in JSON format. Be sure to handle errors properly by checking status codes.

API Key Authentication

Example uses API keys to authenticate requests. Include your API key in the Authorization header of every request.

Obtaining an API Key

You can generate an API key from your dashboard after signing up. Each key is unique and should be stored securely.

Secure Your API Key

Never expose your API key in client-side code or commit it to version control. Use environment variables instead.

Using Your API Key

Include your API key in the Authorization header using Bearer authentication: 

Authorization: Bearer YOUR_API_KEY

Here's a complete example: 

curl -X GET "https://api.example.com/v1/profile" \
  -H "Authorization: Bearer sk_live_1234567890abcdef" \
  -H "Content-Type: application/json"

OAuth 2.0 Authentication

For applications that need to act on behalf of users, Example supports OAuth 2.0 authentication flow.

OAuth Flow

  1. Redirect user to authorization URL
  2. User grants permission
  3. Receive authorization code
  4. Exchange code for access token
  5. Use access token to make API requests

Step 1: Request Authorization

https://api.example.com/oauth/authorize?
  client_id=YOUR_CLIENT_ID&
  redirect_uri=YOUR_REDIRECT_URI&
  response_type=code&
  scope=read write

Step 2: Exchange Code for Token

curl -X POST "https://api.example.com/oauth/token" \
  -d "grant_type=authorization_code" \
  -d "code=AUTH_CODE" \
  -d "client_id=YOUR_CLIENT_ID" \
  -d "client_secret=YOUR_CLIENT_SECRET"
GET

Get All Users

Retrieve a paginated list of all users in your organization.

GET /v1/users

Query Parameters

Parameter Type Description
page integer Page number (default: 1)
limit integer Items per page (default: 20, max: 100)
search string Search query to filter users

Example Request

curl -X GET "https://api.example.com/v1/users?page=1&limit=10" \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json"

POST

Create a User

Create a new user in your organization.

POST /v1/users

Request Body

Field Type Required Description
email string Yes User's email address
name string Yes User's full name
password string Yes User's password (min. 8 characters)
role string No User's role (default: "user")

Example Request

curl -X POST "https://api.example.com/v1/users" \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "email": "newuser@example.com",
    "name": "Alice Johnson",
    "password": "securepassword123",
    "role": "user"
  }'

Example Response

{
  "id": "usr_789ghi",
  "email": "newuser@example.com",
  "name": "Alice Johnson",
  "role": "user",
  "created_at": "2024-10-05T16:45:00Z",
  "message": "User created successfully"
}

Error Codes

Example uses standard HTTP status codes to indicate the success or failure of API requests.

HTTP Status Codes

Code Status Description
200 OK Request completed successfully
201 Created Resource created successfully
400 Bad Request Invalid request parameters
401 Unauthorized Missing or invalid authentication
403 Forbidden Insufficient permissions
404 Not Found Resource does not exist
429 Too Many Requests Rate limit exceeded
500 Internal Server Error Server error occurred

Error Response Format

All error responses have a consistent JSON structure: 

{
  "error": {
    "code": "validation_error",
    "message": "Invalid email format",
    "details": [
      {
        "field": "email",
        "issue": "Must be a valid email address"
      }
    ]
  }
}

Common Error Codes

validation_error

Request validation failed. Check the details field for specific issues.

authentication_error

Authentication credentials are missing or invalid.

rate_limit_error

Too many requests. Please wait before making another request.

resource_not_found

The requested resource was not found.

Rate Limiting

To ensure fair usage and system stability, Example implements rate limits on all API endpoints.

Default Rate Limits

  • Free Tier: 100 requests per hour
  • Pro Tier: 1,000 requests per hour
  • Enterprise Tier: Custom limits

Rate Limit Headers

Every API response includes headers with rate limit information:

Header Description
X-RateLimit-Limit Maximum number of requests allowed per hour
X-RateLimit-Remaining Remaining requests in current window
X-RateLimit-Reset Unix timestamp when the limit will reset

Example Response Headers

X-RateLimit-Limit: 1000
X-RateLimit-Remaining: 847
X-RateLimit-Reset: 1728144000

Handling Rate Limits

When you exceed the rate limit, the API returns a 429 status code: 

{
  "error": {
    "code": "rate_limit_error",
    "message": "Rate limit exceeded. Please try again later.",
    "retry_after": 3600
  }
}

Best Practices

  • • Monitor rate limit headers in responses
  • • Implement exponential backoff when approaching limits
  • • Cache responses where appropriate
  • • Contact support for higher limits if needed
